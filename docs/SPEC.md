# promptward -- specification

Status: finalized for MVP build. This is the source of truth. Update it when a decision changes, then implement.

## 1. Objective

promptward is a drop-in LLM gateway that sits in front of OpenAI/Anthropic and does four things on every call: detects prompt injection inbound, detects secret/PII/data exfiltration both directions, validates structured output against the caller's JSON Schema, and meters token cost per request. The differentiator is an eval harness that PROVES the detection rate per attack class instead of asserting it.

Users: teams shipping LLM features who need a fast, self-hostable checkpoint for the two failure modes that actually hurt in production (untrusted text steering the model; sensitive data leaving in a prompt or response) and who want a measured handle on both plus cost.

Success: point an OpenAI or Anthropic SDK at promptward (change `baseURL` only), keep wire compatibility, and get inline injection/exfil detection, schema-validated output, and per-call cost -- with a `pnpm eval` run that emits the precision/recall/F1 numbers the README cites. Every numeric claim traces to a measured run.

## 2. Tech stack (pinned)

- Detection core: Rust (edition 2021). Hot-path scanners. Deps kept lean: `aho-corasick`, `regex`, `once_cell`, `serde`/`serde_json`, `unicode-normalization`.
- Rust <-> TS binding: `napi-rs` (`napi`, `napi-derive`) + `@napi-rs/cli`. The native addon is the ONLY boundary. TS types are generated by `@napi-rs/cli` (`index.d.ts`) from the Rust types -- no separate type-export tool. (specta was evaluated and dropped: napi already emits synced types for everything crossing the FFI boundary, so it would be a redundant second generator; revisit only if Rust should own the broader findings/event schema. wasm was the alternative FFI, deferred -- it would unlock the manifest-v3 browser stretch.)
- Gateway: TypeScript, Node 24, ESM. HTTP framework `hono` (light, fast, edge-friendly). Schema validation `ajv` (2020-12). Provider calls go out via raw `fetch` to keep the proxy wire-compatible (the inbound body is forwarded as-is); the vendor SDKs are intentionally not used.
- Event store: Postgres (via `pg`), behind a thin store interface with an in-memory implementation for tests and DB-free local runs.
- Dashboard: React + Vite + TypeScript. Dense, technical, dark, severity-coded.
- Workspace: pnpm workspace (`apps/*`, `evals`, `crates/tripwire-core`) + a Cargo workspace (`crates/tripwire-core`).
- Test: `cargo test` (Rust units, TDD), `vitest` (TS).

Pin exact model ids at build time. The cost model and the optional judge use the ids in section 8.

## 3. Architecture

```
SDK (baseURL -> promptward)
        |
   [ gateway ]  TS proxy, wire-compatible with OpenAI /v1/chat/completions + Anthropic /v1/messages
        |  1. scan inbound   -> tripwire-core (napi)   injection + exfiltration + smuggling
        |  2. policy gate    -> allow / redact / block
        |  3. provider call  -> Anthropic / OpenAI   (only if not blocked)
        |  4. validate       -> structured output vs caller JSON Schema (bounded retry)
        |  5. scan outbound  -> tripwire-core (napi)   exfiltration + markdown-exfil
        |  6. record         -> tokens + cost + findings  (event store)
        v
   [ dashboard ]  React: request log, findings, cost
```

The eval path is DB-free and provider-free for the core detector metrics: it loads the labeled datasets, runs `scan` via the napi binding, and computes per-class precision/recall/F1. The gateway, event store, and dashboard are downstream of that proof.

## 4. tripwire-core <-> gateway binding (napi-rs)

- One crate, `crates/tripwire-core`. Pure scanner logic lives in modules (`rlib`, unit-tested by `cargo test`, no napi dependency in the tested path). A thin `#[napi]` wrapper at the crate root, gated behind a `node` feature, exposes the boundary and builds the `cdylib` addon.
- The crate doubles as the npm package `@promptward/tripwire` (napi-rs convention: `package.json` beside `Cargo.toml`). `@napi-rs/cli` builds the `.node` binary and generates `index.d.ts`.
- Boundary signature (stable contract):
  ```
  scan(text: string, direction: "inbound" | "outbound", source?: Source): Finding[]
  ```
  `Finding`, `Kind`, `Severity`, `Source` are `#[napi(object)]` / `#[napi]` types; the generated `index.d.ts` is the single TS definition. Gateway and eval `import { scan, Finding } from "@promptward/tripwire"`.
- Determinism: `scan` is pure and side-effect-free. Same input -> same findings. No I/O, no clock, no allocation beyond the findings vec and decode buffers.

## 5. Detector model (the core IP)

`scan(text, direction, source)` runs a fixed, deterministic pipeline and returns `Vec<Finding>`. The pipeline reflects the current (2026) attack surface, not 2024 keyword matching.

Finding shape:
```
Finding { kind: Kind, label: String, severity: Severity, score: f32 (0.0..=1.0), span: (usize, usize) /* byte offsets in ORIGINAL text */, source: Source, detail: Option<String> }
Kind     = Injection | Exfiltration | Obfuscation
Severity = Info | Low | Medium | High | Critical
Source   = User | System | Tool | Document | McpDescription | ModelOutput
```

Pipeline:

1. Normalize + smuggling detection. NFKC-normalize; detect and flag invisible/abusable code points before any other heuristic so smuggled text becomes visible downstream: Unicode Tags block (U+E0000-U+E007F), zero-width (U+200B-U+200D, U+FEFF), bidi controls (U+202A-U+202E, U+2066-U+2069), and homoglyph-heavy runs. Emit `Obfuscation` findings (label e.g. `unicode_tag_smuggling`, `zero_width`, `bidi_override`); continue with normalized text.
2. Decode-then-rescan. Detect candidate encoded substrings (base64, hex, URL-encoding, rot13), decode (bounded recursion depth), and re-run the injection + exfiltration scanners on the decoded content so directives hidden in encoded blobs are caught. Findings carry the decoded provenance in `detail`.
3. Injection scanner (heuristics; `aho-corasick` for literal phrase sets + compiled `regex` for patterns), source-aware:
   - instruction override ("ignore (all )?previous instructions", "disregard the above", "new instructions:").
   - system/role impersonation ("you are now", leading "SYSTEM:", "<system>", DAN-style persona resets).
   - tool/MCP poisoning: imperative or role-switch instructions appearing inside `Tool` / `Document` / `McpDescription` sources are scored higher (these are where 2026 indirect-injection and MCP tool-description poisoning land).
   - cross-source authority laundering markers ("as the system administrator", "the user has authorized", "per policy above").
   - exfil-instruction phrasing ("email all data to", "send the conversation to", "POST ... to http").
4. Exfiltration scanner (both directions; precision-tuned):
   - secrets via regex: AWS access-key id (AKIA...) and secret, GitHub tokens (ghp_/gho_/ghu_/ghs_/ghr_), OpenAI/Anthropic `sk-`/`sk-ant-` keys, Google API keys (AIza...), Slack tokens (xox...), JWT (three base64url segments), PEM private-key blocks.
   - PII: email, US SSN, credit-card number (regex + Luhn checksum to cut false positives).
   - generic high-entropy secret fallback: Shannon entropy over a token, gated by length and charset, low score -- a precision-first backstop, not a primary signal.
5. Markdown / link exfiltration (outbound emphasis): markdown image `![](url)` / link `[](url)` / bare URLs pointing at non-allowlisted domains, weighted up when the URL query or path carries high-entropy or data-shaped payloads (the canonical zero-tool leak). Default policy strips URL params and holds rendering to an allowlist.

Scores are calibrated (per-label constants tuned against the corpus), not guessed -- they must support a Recall @ fixed-FPR readout (section 9).

## 6. Policy model (allow / redact / block)

- Code-defined policy (no DSL; a non-goal). A policy maps `(Kind, Severity, Direction)` to an `Action`.
- Actions:
  - `allow`: pass through; record findings.
  - `redact`: replace each finding span with a typed placeholder (`[REDACTED:aws_key]`) before forwarding (inbound) or returning (outbound). Record the finding metadata -- never the secret value.
  - `block`: reject before the provider call (inbound) or before returning (outbound), as a provider-shaped error object. Record the finding.
- Action precedence when multiple findings fire: `block` > `redact` > `allow` (most severe wins).
- Redaction is surfaced honestly: redacted responses carry a `promptward` metadata field (and response header) listing what was redacted by label and span, so the caller knows the body was modified.
- Default policy (overridable): inbound injection High/Critical -> block, Low/Medium -> flag(allow+record); inbound exfiltration -> redact; outbound exfiltration + markdown-exfil -> redact (block on Critical).
- Multi-turn: detection state can persist per conversation (the 2026 multi-turn / authority-laundering finding) -- the store keys findings by conversation id. MVP enforces per-request; session-aware escalation is a documented seam, not required for the first eval.

## 7. Structured-output validation + retry

- If the caller supplies a JSON Schema (OpenAI `response_format: { type: "json_schema", json_schema: { schema } }`, or Anthropic via a tool / output schema), promptward validates the provider's output against it with `ajv` (strict, 2020-12).
- On validation failure: bounded retry (default `maxRetries = 2`) re-issuing the call with a corrective instruction appended ("Your previous output failed schema validation: <ajv errors>. Return only valid JSON matching the schema."). Each attempt's tokens are metered.
- On exhausted retries: surface the failure honestly -- return a provider-shaped error indicating schema validation failed after N attempts, including the ajv errors. Never silently return invalid output. Record the failure, retry count, and cumulative cost.

## 8. Cost model (pinned ids)

Cost = `input_tokens/1e6 * input_price + output_tokens/1e6 * output_price`, summed across all attempts (retries + optional judge). Price table (USD per 1M tokens), pinned at build time:

| Model | id | input | output |
|---|---|---|---|
| Opus 4.8 | `claude-opus-4-8` | 5.00 | 25.00 |
| Sonnet 4.6 | `claude-sonnet-4-6` | 3.00 | 15.00 |
| Haiku 4.5 | `claude-haiku-4-5` | 1.00 | 5.00 |
| Fable 5 | `claude-fable-5` | 10.00 | 50.00 |

OpenAI ids/prices are added to the same table as the OpenAI provider path lands. Unknown model id -> cost recorded as `null` with a `model_unpriced` flag (never a fabricated number).

## 9. Eval contract

The eval is the credibility centerpiece. It is deterministic and reproducible; its output populates the README.

- Dataset: versioned JSONL under `evals/datasets/`. Record schema:
  ```
  { "id": string, "text": string, "label": "injection" | "exfiltration" | "benign", "bucket": string, "direction"?: "inbound" | "outbound", "note"?: string }
  ```
  `label` is the ground-truth class the scanner must detect; `bucket` is the finer attack class for per-class reporting.
- Buckets (named coverage): `direct_injection`, `indirect_injection`, `tool_result_injection`, `mcp_description_poisoning`, `unicode_smuggling`, `base64_wrapped`, `markdown_image_exfil`, `secret_leak`, `pii_leak`, plus `benign` and `benign_hard_negative` (injection-shaped but legitimate: security questions, quoted attack text, "explain prompt injection", code with a variable named `password`).
- Scoring: for an example labeled injection|exfiltration, true positive if `scan` returns a finding of the matching `Kind`; a benign example with any finding is a false positive. Report per-bucket and overall precision / recall / F1, a confusion matrix, and -- the credible 2026 metric -- Recall @ 1% FPR (using calibrated finding scores against the benign + hard-negative set). Also report the benign false-positive rate explicitly (utility-under-attack proxy: detection must not break normal use).
- Determinism: fixed dataset version, pinned model ids, cached provider responses (judge only), single `pnpm eval` entrypoint that prints the table and writes a JSON artifact the README and dashboard read. No network needed for the core detector metrics.
- Caveat documented in the eval output: static datasets overstate robustness (the AgentDyn point). Include a small adaptive/red-team slice where feasible so the claimed rate is not only against a frozen corpus, and say so.

## 10. LLM-judge (opt-in)

- An optional second-stage classifier for fuzzy injection cases the deterministic scanner is unsure about. Off by default (the core metrics stay deterministic and free).
- Model: `claude-haiku-4-5` (cheapest at 1.00/5.00 per 1M, 200K context). Cached by input hash, cost-tracked, small `max_tokens`, structured verdict.
- When enabled in the gateway it augments scanner findings; when enabled in the eval it is reported as a separate, clearly-labeled run with its cost.

## 13. Code style

- Rust: pure, deterministic, allocation-light. Compile every `Regex`/`AhoCorasick` once via `once_cell::sync::Lazy`. Never panic on input -- user text never reaches `unwrap`/`expect`; on any internal failure return what was found so far. Spans are byte offsets into the original text. The `#[napi]` wrapper is a thin shim over the pure functions.
  ```rust
  static OVERRIDE: Lazy<AhoCorasick> = Lazy::new(|| AhoCorasick::new(OVERRIDE_PHRASES).unwrap());
  pub fn scan(text: &str, direction: Direction, source: Source) -> Vec<Finding> { /* pure */ }
  ```
- TypeScript: strict mode, ESM, no `any`. Import the scanner and its types from `@promptward/tripwire`; validate with `ajv`; handle provider and validation errors explicitly in the provider's wire shape.
  ```ts
  import { scan, type Finding } from "@promptward/tripwire";
  const findings: Finding[] = scan(text, "inbound");
  ```
- ASCII only everywhere (`--` not em-dash, straight quotes, `->` not arrow).

## 14. Testing strategy

- Rust units (`cargo test`, TDD): write failing tests from the labeled corpus first, then implement each scanner to green. Table tests per detector (positive patterns + hard negatives) plus dataset-derived fixtures. This is the red-green driver for tripwire-core.
- Eval (`pnpm eval`): the integration-level proof. Runs the built scanners over the full corpus via napi and produces the metrics. Deterministic; runnable in CI as a gate (fail if F1 or Recall@1%FPR regresses below a pinned floor).
- TS units (`vitest`): gateway pipeline against a mocked provider (policy actions, redaction representation, schema-validation retry/exhaustion, cost summation), and eval-runner determinism (same input -> same numbers).

## 15. Boundaries

- Always: run `cargo test` and `pnpm test` before a commit; keep committed files ASCII-only; derive every eval number from a measured run; pin model ids; commit at green points.
- Ask first: adding any dependency beyond the pinned set in section 2; event-store schema changes; anything outward-facing (creating the GitHub repo, pushing).
- Never: commit secrets; reference AI/agents/assistant tooling or internal build notes in committed files or commit messages; hand-write or estimate an eval number; push to GitHub without explicit approval.

## 16. Success criteria

- `pnpm eval` runs deterministically with no network for the core metrics and emits a per-bucket precision/recall/F1 table, a confusion matrix, Recall @ 1% FPR, and the benign false-positive rate; the README rate table is generated from that artifact.
- tripwire-core passes `cargo test` over the labeled corpus with coverage of every named bucket; `scan` is pure and deterministic.
- The gateway is wire-compatible (OpenAI `/v1/chat/completions`, Anthropic `/v1/messages`), runs scan -> policy -> provider -> schema-validate(+retry) -> scan -> record, and reports per-request tokens and cost.
- The dashboard renders the request log, findings (severity-coded), and cost from the event store.
- Every numeric or rate claim in the README traces to the eval artifact.

## 17. Positioning (honest; for README framing)

- vs Lakera Guard (closed, per-call, fast, high accuracy): promptward's edge is open + self-hostable + measured-in-public; do not claim to beat its accuracy without a measured run.
- vs LLM Guard (MIT scanner library): closest peer. Honest framing -- same scanner spirit, but with a published per-attack-class eval that proves the rate, a Rust hot path for low-latency inline scanning, and a built-in cost meter. LLM Guard is a scanner library, not a gateway with a proof and a cost meter.
- vs Promptfoo (CI red-teaming): complementary, not competing. Promptfoo proves quality in CI; promptward enforces at runtime and ships its own eval.
- vs LiteLLM (the dominant OpenAI-format gateway): do not try to out-gateway it. Lead with security-as-the-product where its guardrails are a thin bolt-on, and stay wire-compatible so promptward can sit in front of or beside it.
- vs CaMeL / LlamaFirewall (research-grade, design-level and model-based defenses): concede they are stronger research artifacts; promptward's niche is the pragmatic, drop-in, OpenAI/Anthropic-compatible box that bundles fast pattern detection + structured-output validation + cost metering + a reproducible eval in one MIT package.

## 18. Non-goals

- A full SIEM / analytics backend. promptward emits events; it is not the log pipeline.
- Model hosting or fine-tuning. It proxies hosted providers.
- A policy DSL. Policies are code-defined; generalize only if needed.
- Beating closed commercial detectors on accuracy without a measured run to back it.

## 19. Open questions

- Provider-first order: Anthropic path first (the pinned model set is Anthropic), OpenAI path second. Wire-compat tests cover both request shapes regardless.
- Recall @ 1% FPR requires calibrated per-label scores; if calibration proves noisy on the small corpus, report it with the corpus size stated and treat the floor as provisional until the dataset grows.
- Browser-extension stretch would favor a wasm build of the same core; the pure-module split in section 4 keeps that a small addition rather than a refactor.
